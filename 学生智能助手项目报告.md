## 一、项目概述

### 1.1 项目背景

留学教育机构的学生服务长期依赖人工处理：请假靠纸质审批、投诉靠微信沟通、心理关怀靠老师直觉、申请进度靠学生反复催问。存在以下痛点：

- **服务碎片化**：请假、投诉、心理、学业各走各的渠道
- **审批滞后**：班主任不在办公室就无法审批
- **心理风险不可见**：学生情绪波动完全依赖老师个人判断
- **信息不透明**：学生不知道申请到哪一步、DDL 是否临近

### 1.2 模块定位

学生智能助手是 5 大业务模块之一，面向**在读学生**和**班主任**，提供 8 大业务场景的 AI 驱动服务闭环。

```
学生侧：多轮对话入口 → AI 理解意图 → 自动执行/引导
教师侧：REST API → 待办列表 → 在线审批 → 通知回调
```

## 四、各业务实现方式

### 4.1 请假审批闭环

```
学生: "感冒了请假三天"
  → POST /student/chat
    → LLM 意图识别 → leave_request
    → LLM 槽位提取: leave_type=sick, start=明天, end=后天, reason=感冒
    → 信息完整 → 回显确认
  → "确认" → POST /student/leave-requests
    → 双重逻辑外键校验（sys_user + student_info）
    → 写入 student_admin_service 表（status=pending）
    → 推送通知至班主任（notify_type=leave_submitted）

教师: GET /student/leave-requests?status=pending → 待审批列表
  → PUT /student/leave-requests/{id}/approve
    → 角色校验（仅 employee/admin）
    → 条件更新 WHERE status='pending'（防并发）
    → 推送通知至学生（notify_type=leave_approved/rejected）
```

### 4.2 心理关怀与预警

```
学生: "最近压力好大睡不着"
  → POST /student/chat
    → LLM 意图识别 → psych_express
    → LLM 共情回复（不限轮次，上下文保持）
    → 后台静默写入 psych_record

系统自动判定:
  - 分值 < 20 或关键词匹配 → high → 创建预警 + 通知班主任 + 通知学生
  - 分值 20-39 → medium → 创建预警 + 通知班主任
  - 分值 ≥ 40 → low → 仅更新画像
```

**三重防幻觉机制**：
1. 心理回复用 LLM 生成，但**不提及"预警"字眼**，学生无感知
2. 预警判定在后端做（关键词 + 分值规则），不依赖 LLM 判断
3. 高危关键词硬编码白名单，保守阈值宁可多报不可漏报

### 4.3 投诉反馈闭环

```
学生: "签证太慢了投诉"
  → POST /student/chat
    → LLM 意图识别 → complaint_submit
    → LLM 槽位提取: ticket_type=complaint, category=签证办理
    → 自动判定优先级（退款/律师 → urgent）
  → "确认" → POST /student/feedback-tickets
    → 写入 student_feedback_ticket 表

教师: PUT /student/feedback-tickets/{id}
  → 状态流转: pending → processing → resolved → closed
  → resolved/closed 时推送通知至学生
```

### 4.4 查询类（成绩/DDL/申请/通知）

**零 LLM 参与回复生成**，全部使用模板格式化数据库查询结果，彻底杜绝 AI 幻觉：

| 意图 | 关键词 | API 调用 | 回复方式 |
|------|------|---------|:--:|
| score_query | 成绩/GPA/多少分/绩点 | GET /student/scores | 模板 |
| deadline_query | DDL/截止/考试 | GET /student/deadlines | 模板 |
| application_progress | 申请/offer/offer/进度 | GET /student/applications | 模板 |
| notification_query | 通知/消息/小红点 | GET /student/notifications | 模板 |

### 4.5 多轮对话引擎

```
POST /student/chat?student_id=10&message=...
        ↓
┌──────────────────────────────────────┐
│         ConversationState            │
│  student_id, intent, filled_slots,   │
│  missing_slots, history, confirm     │
└──────────────────────────────────────┘
        ↓
  LLM 意图识别 + 槽位提取（JSON）
        ↓
  关键词兜底（LLM 调用失败时自动降级）
        ↓
┌──────────────────────────────────┐
│  10 路意图路由                    │
│  请假 / 投诉 / 心理 / 申请 /     │
│  DDL / 成绩 / 通知 / 情绪 /      │
│  增值 / 闲聊 / 未知兜底          │
└──────────────────────────────────┘
        ↓
  槽位管理：已填 / 缺失 → 追问 / 确认 / 执行
```

**对话状态**：Python 内存 dict，5 分钟超时自动清理，意图切换自动重置槽位。

## 七、难点与解决方案

### 难点 1：无物理外键策略 → 应用层数据一致性

**问题**：数据库全面禁用 FOREIGN KEY，所有表间关系靠应用层维护。

**解决**：
- 所有 `{entity}_id` 写入前通过 `SELECT EXISTS(...)` 校验引用记录存在
- 索引替代物理外键（所有外键字段建立普通索引）
- 条件 UPDATE（`WHERE status=X`）替代数据库约束实现状态机

### 难点 2：AI 幻觉 → 查询类零 LLM 回复

**问题**：LLM 查询成绩时可能编造"MIT录取"、"85分"等虚假数据。

**解决**：
- 成绩/DDL/申请/通知 4 类查询**不经过 LLM 生成回复**
- 使用 Python 模板直接格式化数据库真实数据
- LLM 仅用于意图识别和槽位提取，不参与回复生成

### 难点 3：Dify Chatflow 沙箱故障 → 纯 Python 方案

**问题**：Dify Sandbox 容器网络隔离、SSRF 代理拦截，代码节点持续报错。

**解决**：
- 完全抛弃 Dify Chatflow，用 FastAPI 原生路由实现对话引擎
- 意图识别改用通义千问 API 直调
- 对话状态用 Python 内存 dict 管理
- 回复用模板 + LLM 双模式，模板保底防幻觉

### 难点 4：多轮槽位收集 → 状态机 + 意图切换

**问题**：请假需要 4 个字段，投诉需要 3 个字段，学生可能中途改变主意。

**解决**：
- `ConversationState` 数据类持有 intent + filled_slots + missing_slots
- 每次对话从 LLM JSON 响应提取槽位值
- 意图切换时自动重置槽位
- 确认/取消关键词精匹配防误触

### 难点 5：ORM 与 db_init.sql 字段不一致

**问题**：模型严格按照《数据库设计规范 V2.1》编写，但实际 `db_init.sql` 字段名不同（如 `current_stage` vs `stage`，`deadline_date` vs `deadline`）。

**解决**：
- 以 `db_init.sql`（实际数据库）为唯一真相源
- 逐一修正 3 张表的字段名和类型
- 删除不存在列（如 `student_score.exam_date`）

## 八、亮点总结

### 1. 审批闭环全链路

"学生提交 → 通知推送 → 教师审批 → 结果回调"完整闭环，涉及 2 张业务表 + 1 张通知表 + 4 条推送路径，全部在 Service 层单事务完成。

### 2. 心理预警静默机制

学生在对话中完全不知道预警被触发，后台自动判定风险等级、创建预警工单、推送给班主任。既保护学生隐私，又确保高危情况不遗漏。

### 3. 双重意图识别（LLM + 关键词兜底）

LLM 调用失败时自动降级到关键词匹配，确保对话引擎永不崩溃。18 个关键词 + 路径权重匹配 + 疾病症状自动判定病假类型。

### 4. 零 AI 幻觉

所有数据查询类回复用模板格式化，不经过 LLM。只有闲聊和槽位追问才调 LLM，将幻觉风险降到最低。

### 5. 教师端零开发对接

教师组员只需调 6 个 REST API 即可完成全部业务，无需理解学生对话引擎、LLM 调用、槽位管理等复杂逻辑。

### 6. 对话/API 双入口

同一业务既可以通过自然语言对话完成（学生侧），也可以通过 REST API 调用（教师/系统侧），两套入口共享同一 Service 层。
